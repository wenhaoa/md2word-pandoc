# 首次安装指南

## 1. 环境依赖

本 Skill 需要以下软件环境，请先确认已安装：

| 软件    | 最低版本 | 安装命令 / 获取方式                                                 |
| ------- | -------- | ------------------------------------------------------------------- |
| Pandoc  | 2.x      | `winget install JohnMacFarlane.Pandoc` 或从 https://pandoc.org 下载 |
| Node.js | 14.x     | `winget install OpenJS.NodeJS.LTS` 或从 https://nodejs.org 下载     |

安装验证：
```powershell
pandoc --version
node -v
```

## 2. 配套 Rules（推荐添加到 GEMINI.md）

本 Skill 配合以下 Markdown 转 Word 写作规范使用效果最佳。建议将以下内容追加到用户的全局规则文件 `~/.gemini/GEMINI.md` 中：

```markdown
# Part D: Markdown 转 Word 规范

**适用范围**：仅当用户明确要求"转Word/转docx"时才适用以下规则。普通 Markdown 文档（README、笔记、工作方案等）不受这些限制，可自由使用加粗、`> [!NOTE]`、Mermaid 代码块、公式包裹等。

生成需要转换为 Word 的 Markdown 文档时，必须遵守以下规则（配合 `md2word-pandoc` 技能）。

## D.1 文档头部

- 使用 YAML frontmatter 设置文档标题，**禁止**使用 `#` H1 标题
- `title` 字段**必须填写**，用于生成 Word 封面页标题（通过 `{{TITLE}}` 占位符注入），不写则封面标题为空
- **禁止**使用 `---` 水平分隔线（会产生多余横线或分页）

## D.2 章节编号格式

| 标题级别 | 格式         | 示例                  |
| -------- | ------------ | --------------------- |
| 一级     | `## N.`      | `## 1. 软件概述`      |
| 二级     | `### N.N`    | `### 1.1 软件用途`    |
| 三级     | `#### N.N.N` | `#### 4.1.1 场景概述` |
| 附录     | `## 附录 X`  | `## 附录 A 术语表`    |

- 一级标题数字后**必须带点号**（`1.`），确保过滤器可靠清除
- **禁止** `## 1 标题` 格式（数字+空格，Pandoc 拆分后清除不稳定）

## D.3 正文格式

- 避免不必要的 `**加粗**`，尤其是列表项和表格内容
- 表格正常使用（过滤器自动应用 `TableContent` 样式）
- 代码块、引用块正常使用

## D.4 Markdown 预检清单（AI 在生成/修改报告时自动检查）

- 图片引用编号格式：图 N-M（N=章节号，M=章内序号），按出现顺序连续编号
- 图片标题行后必须空一行
- 表格前后必须各空一行
- 行内公式使用 `$...$`，独立公式使用 `$$...$$`
- **简单数字+单位**（如1100km、650kg、500m）**禁止**用公式包裹，直接写纯文本；仅对含运算符、上下标、希腊字母等**真正的数学表达式**使用公式
- 中文与英文/数字之间不需额外空格（与 Pandoc 转换兼容）
- 标题行前空两行，后空一行
- **禁止**使用 GitHub 风格提示块（`> [!NOTE]`、`> [!WARNING]` 等），Pandoc 无法识别，改用 "注：" 前缀的普通段落
- **Mermaid 图表**必须先通过 `mmdc`（mermaid-cli）转为 PNG 图片，再以 `![](images/xxx.png)` 引用，**禁止**在需要转 Word 的文档中保留 mermaid 代码块

## D.5 列表格式

- 正式报告中**禁止**使用 `-` 无序列表（Pandoc 转 Word 后圆点符号不正规）
- 有序步骤：使用 `1. 2. 3.` 数字编号
- 并列要点：使用手动编号 `（1）（2）（3）`，写成段内文字，用分号分隔
- 少量短要点（≤3项且无标签）：可合并为一段，用分号连接
- 二级子项：使用 `a）b）c）` 或写为段内顿号分隔
- **段落优先原则**：对某主题的多方面描述应写成连贯段落（用"同时"、"此外"、"在……方面"等连接词串联），而非逐条列举。仅当内容属于并列等价项（清单、步骤、指标列表）且读者需逐条确认时才用编号。同一章节内分点列举不宜超过两处。
```

## 3. 配套 Workflow（推荐安装）

建议将以下工作流文件保存为 `~/.gemini/antigravity/global_workflows/report-check.md`：

```markdown
---
description: Markdown 报告图片编号与格式预检，在转 Word 或提交前运行
---

# 报告格式预检工作流

## 1. 扫描目标文件
确认要检查的 Markdown 报告文件路径。如果用户未指定，扫描当前工作目录下的所有 `.md` 文件。

## 2. 图片编号连续性检查
- 提取所有 `图 N-M` 格式的引用
- 验证每章内编号从 1 开始且连续（图 1-1, 1-2, 1-3...）
- 报告不连续或缺失的编号

## 3. 格式合规检查
逐项检查以下规则（对应 Rules D.4）：

- [ ] 图片标题行后是否有空行
- [ ] 表格前后是否各有空行
- [ ] 行内公式是否使用 `$...$` 格式
- [ ] 数字+单位是否使用行内公式（如 `$500 \, \text{km}$`）
- [ ] 标题行前是否空两行、后是否空一行
- [ ] 是否存在不必要的 CJK-Latin 额外空格

## 4. 输出检查报告
以表格形式汇总所有问题：

| 行号 | 问题类型 | 当前内容 | 建议修改 |
| ---- | -------- | -------- | -------- |

## 5. 自动修复（可选）
如果用户确认，自动修复可以安全处理的格式问题（如空行、空格）。
图片编号重排需要用户确认后再执行。
```

## 4. 快速部署清单

收到本 Skill 文件夹后，按以下顺序操作：

1. 将整个 `md2word-pandoc` 文件夹复制到 `~/.gemini/antigravity/skills/`
2. 安装 Pandoc 和 Node.js（见第1节）
3. 将第2节的 Rules 内容追加到 `~/.gemini/GEMINI.md`（如果文件不存在需新建）
4. 将第3节的 Workflow 文件保存到 `~/.gemini/antigravity/global_workflows/report-check.md`
5. （可选）配置 PowerShell `md2word` 快捷命令（见 SKILL.md 快速开始章节）

完成后即可使用 `md2word "文件.md"` 或让 AI 执行"转 Word"。
